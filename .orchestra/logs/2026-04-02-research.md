## Iteration 4: Provocation

**Provocation type:** Inversion — "Are quality scores measuring quality or GitHub popularity?"

**Analysis:**
The quality score formula (`compute_quality_score` in db.py:1983) uses:
- Completeness: description length, tags, maker_id, tagline, source_type, integration_python
- Engagement: view_count, click_count, mcp_view_count, upvotes, reviews
- Health: dead/degraded/alive multiplier

**Finding:** GitHub stars are NOT in the quality score at all. The formula is engagement-native. Good.

**Potential issue identified:** `integration_python` (+0.15 completeness bonus) is a manually-curated admin field. Only ~50-100 tools have it set. This means the quality score has a ceiling of ~85/100 for uncurated tools vs 100/100 for admin-curated ones.

**More important inconsistency:** `install_command` absence gets a -40 point search penalty but is NOT factored into quality_score. A tool without install_command ranks lower in search (correct) but can still show a high quality_score on its profile page (confusing).

**Action taken:** None — this is a design decision, not a bug. The search penalty and quality score serve different purposes. Logging as observation.
